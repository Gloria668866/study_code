# CI/CD中的测试策略

## 持续集成中的测试分层

在CI/CD流水线中，测试按执行速度和覆盖范围分为多个层级，每一层在不同阶段执行：

### 1. 单元测试（Unit Tests）
- **执行时机**：代码提交后立即触发
- **执行速度**：毫秒到秒级
- **覆盖范围**：单个函数或方法
- **工具**：JUnit、Mockito、TestNG、pytest
- **要求**：所有单元测试必须在代码合并前通过

### 2. 组件测试（Component Tests）
- **执行时机**：单元测试通过后
- **覆盖范围**：模块级别，测试多个类或模块的协作
- **特点**：可能涉及真实的数据库或外部依赖的Mock

### 3. 集成测试（Integration Tests）
- **执行时机**：组件测试通过后
- **覆盖范围**：多个服务之间的接口交互
- **工具**：Postman/Newman、REST Assured、Testcontainers
- **重点**：验证API契约、数据库交互、消息队列通信

### 4. 端到端测试（E2E Tests）
- **执行时机**：部署到预发布环境后
- **覆盖范围**：完整的用户操作流程
- **工具**：Selenium、Cypress、Playwright
- **注意**：E2E测试不稳定，应控制在关键业务流程

### 5. 性能测试（Performance Tests）
- **执行时机**：E2E测试通过后，或定期执行
- **类型**：负载测试、压力测试、浸泡测试
- **工具**：JMeter、Gatling、k6、Locust

## 测试左移与测试右移

### 测试左移（Shift Left）
将测试活动提前到开发周期的更早阶段，在编码阶段就发现和修复缺陷。

实践方法：
- 需求评审阶段就编写验收测试用例
- 开发人员编写单元测试（TDD/BDD）
- Code Review中加入测试检查清单
- 静态代码分析（SonarQube）集成到IDE中

### 测试右移（Shift Right）
在生产环境中持续进行测试和监控：

- **A/B测试**：比较新旧版本的用户体验
- **金丝雀发布**：小比例用户先体验新版本
- **混沌工程**：主动注入故障验证系统韧性
- **生产监控与告警**：APM工具持续采集数据

## 自动化测试流水线设计

典型的Jenkins Pipeline阶段：

```
Checkout -> Build -> Unit Test -> Static Analysis ->
Integration Test -> Deploy to Staging -> E2E Test ->
Performance Test -> Deploy to Production -> Smoke Test
```

### 关键实践

1. **快速反馈**：单元测试应在5分钟内完成
2. **失败即停止**：任何阶段失败都阻止后续阶段
3. **并行执行**：将测试用例拆分为多个Job并行运行
4. **测试数据管理**：使用容器化数据库（Testcontainers）确保测试环境一致性
5. **测试报告**：生成HTML/XML格式报告便于分析

### 测试覆盖率门禁

在CI中设置覆盖率阈值：
- 行覆盖率：不低于80%
- 分支覆盖率：不低于70%
- 新增代码覆盖率：不低于90%

低于阈值的代码无法合并到主分支。
