# 测试用例设计方法详解：等价类、边界值、场景法

## 一、测试用例设计概述

测试用例设计是软件测试活动中最核心的智力活动，直接影响测试的质量和效率。好的测试用例能用最少的测试数据覆盖最多的业务场景，最大程度地发现缺陷。测试用例设计方法分为黑盒测试设计方法和白盒测试设计方法两大类。

**高质量测试用例的特征**：
- **有效性**：能有效发现程序中的缺陷
- **可重复性**：任何人在任何时候执行都能得到一致结果
- **独立性**：用例之间尽量解耦，减少相互依赖
- **可维护性**：用例结构清晰，易于理解和更新
- **可追踪性**：每个用例都能追溯到具体需求

## 二、等价类划分法

### 2.1 方法原理

等价类划分法（Equivalence Partitioning）的核心思想是将程序的输入域划分为若干子集（等价类），同一个等价类中的任何输入数据对程序的测试效果是等价的。从每个等价类中选取一个或少数几个代表值进行测试，即可代表整个等价类的测试覆盖。

等价类分为两种：
- **有效等价类**：符合程序规格说明的、有意义的合理输入数据的集合
- **无效等价类**：不符合程序规格说明的、无意义的不合理输入数据的集合

### 2.2 划分原则

1. 如果输入条件规定了取值范围，可划分出一个有效等价类和两个无效等价类（小于最小值、大于最大值）
2. 如果输入条件规定了输入值的集合，可划分出一个有效等价类和一个无效等价类
3. 如果输入条件规定了"必须如何"的条件，可划分出一个有效等价类和多个无效等价类
4. 如果输入条件是布尔值，划分出一个有效等价类和一个无效等价类
5. 如果确知等价类中的元素处理方式不同，应进一步细分为更小的等价类

### 2.3 实战示例

**案例：用户注册表单验证**

假设一个注册表单包含以下字段和验证规则：
- 用户名：4-20位字母数字下划线，字母开头
- 密码：8-16位，包含大小写字母和数字
- 年龄：1-120岁整数
- 邮箱：符合邮箱格式

```python
# 等价类分析表

# 用户名等价类：
# 有效等价类1：4-20位，字母开头，仅含字母数字下划线  (如：user_123)
# 有效等价类2：4位边界值，字母开头  (如：abc1)
# 有效等价类3：20位边界值，字母开头  (如：a1234567890123456789)
# 无效等价类1：长度小于4(3位)  (如：abc)
# 无效等价类2：长度大于20(21位)  (如：a加上20个字符)
# 无效等价类3：数字开头  (如：123user)
# 无效等价类4：包含特殊字符  (如：user@name)
# 无效等价类5：空值/未输入
# 无效等价类6：仅空格

# 密码等价类：
# 有效等价类1：8-16位，至少包含大小写字母和数字  (如：Abc12345)
# 有效等价类2：8位边界值  (如：Ab12345c)
# 有效等价类3：16位边界值
# 无效等价类1：长度小于8  (如：Ab12345)
# 无效等价类2：长度大于16
# 无效等价类3：仅包含小写字母和数字，缺少大写  (如：abc12345)
# 无效等价类4：仅包含字母，缺少数字  (如：Abcdefgh)
# 无效等价类5：纯数字

# 年龄等价类：
# 有效等价类1：1-120之间  (如：25)
# 无效等价类1：小于1  (如：0, -5)
# 无效等价类2：大于120  (如：121, 999)
# 无效等价类3：非整数  (如：15.5)
# 无效等价类4：非数字  (如："二十")

# 邮箱等价类：
# 有效等价类1：标准格式  (如：user@example.com)
# 有效等价类2：带子域名  (如：user@mail.example.com)
# 有效等价类3：带加号  (如：user+tag@example.com)
# 无效等价类1：缺少@符号  (如：userexample.com)
# 无效等价类2：缺少域名  (如：user@)
# 无效等价类3：包含空格  (如：user @example.com)
# 无效等价类4：连续的点  (如：user@example..com)
```

## 三、边界值分析法

### 3.1 方法原理

边界值分析法（Boundary Value Analysis, BVA）基于大量的测试实践经验：程序在边界值附近出错的概率远高于内部值。因此，测试的重点应该放在输入域的边界上。

### 3.2 边界值选取规则

对于取值范围[a, b]，应测试以下边界值：
- a-1（略小于最小值）
- a（刚好等于最小值）
- a+1（略大于最小值）
- b-1（略小于最大值）
- b（刚好等于最大值）
- b+1（略大于最大值）

这就是经典的**边界值6点法**。

### 3.3 实战案例

**案例1：订单金额输入框（有效范围0.01-99999.99）**

```python
# 边界值测试数据
test_data = [
    # 金额边界值测试
    {"amount": 0.00, "expected": "fail", "reason": "最小值-1（无效，最小是0.01）"},
    {"amount": 0.01, "expected": "pass", "reason": "刚好最小值（有效）"},
    {"amount": 0.02, "expected": "pass", "reason": "最小值+1（有效）"},
    {"amount": 50000.00, "expected": "pass", "reason": "中间值（有效）"},
    {"amount": 99999.98, "expected": "pass", "reason": "最大值-1（有效）"},
    {"amount": 99999.99, "expected": "pass", "reason": "刚好最大值（有效）"},
    {"amount": 100000.00, "expected": "fail", "reason": "最大值+1（无效）"},
    
    # 精度边界值测试
    {"amount": 0.001, "expected": "fail", "reason": "超过2位小数精度"},
    {"amount": 100.001, "expected": "fail", "reason": "超过2位小数精度"},
    {"amount": 100.999, "expected": "fail", "reason": "超过2位小数精度"},
]
```

**案例2：分页查询参数（page: >=1, pageSize: 1-100）**

```python
# 分页参数的边界值分析
# page边界值：0, 1, 2, Integer.MAX_VALUE
# pageSize边界值：0, 1, 2, 99, 100, 101, -1

page_boundary_test_data = [
    # page边界
    {"page": 0, "pageSize": 10, "expected": "fail", "reason": "page=0（无效）"},
    {"page": -1, "pageSize": 10, "expected": "fail", "reason": "page为负数（无效）"},
    {"page": 1, "pageSize": 10, "expected": "pass", "reason": "page=1（有效最小）"},
    {"page": 2, "pageSize": 10, "expected": "pass", "reason": "page=2（有效）"},
    {"page": 2147483647, "pageSize": 1, "expected": "pass", "reason": "page极大值"},
    
    # pageSize边界
    {"page": 1, "pageSize": 0, "expected": "fail", "reason": "pageSize=0（无效）"},
    {"page": 1, "pageSize": -1, "expected": "fail", "reason": "pageSize为负（无效）"},
    {"page": 1, "pageSize": 1, "expected": "pass", "reason": "pageSize=1（有效最小）"},
    {"page": 1, "pageSize": 2, "expected": "pass", "reason": "pageSize=2（有效）"},
    {"page": 1, "pageSize": 99, "expected": "pass", "reason": "pageSize=99（有效）"},
    {"page": 1, "pageSize": 100, "expected": "pass", "reason": "pageSize=100（有效最大）"},
    {"page": 1, "pageSize": 101, "expected": "fail", "reason": "pageSize=101（无效）"},
]
```

## 四、场景法

### 4.1 方法原理

场景法（Scenario Testing）通过模拟用户真实使用场景来设计测试用例。这种方法基于业务流程，将基本流（正确的主流程）和备选流（各种异常/分支路径）组合成完整的测试场景。

场景法特别适合测试业务流程复杂、有多条操作路径的系统，如电商订单、保险理赔、审批流程等。

### 4.2 场景矩阵设计

**案例：电商下单流程**

```
基本流：
1. 用户浏览商品 → 2. 加入购物车 → 3. 进入结算页 → 4. 选择收货地址
→ 5. 选择支付方式 → 6. 确认订单 → 7. 支付成功 → 8. 下单完成

备选流定义：
B1: 未登录时触发登录
    在步骤3触发，跳转登录页，登录成功后回到结算页

B2: 库存不足
    在步骤2/6触发，提示库存不足

B3: 收货地址为空
    在步骤4触发，引导填写新地址

B4: 优惠券使用
    在步骤6之前触发，选择优惠券

B5: 支付失败
    在步骤7触发，提示支付失败并提供重新支付选项

B6: 订单超时取消
    在步骤6后，30分钟未支付自动取消
```

**场景组合矩阵**：

| 场景编号 | 场景描述 | 包含的流 | 测试数据 |
|---------|---------|---------|---------|
| S1 | 正常下单 | 基本流(1-8) | 已登录用户，有默认地址，商品有库存 |
| S2 | 游客下单 | B1 + 基本流(1-8) | 未登录用户，需要在结算时登录 |
| S3 | 库存不足 | 基本流(1-2) + B2 | 商品仅剩1件，但加入2件 |
| S4 | 新增地址 | 基本流(1-3) + B3 + 基本流(4-8) | 新注册用户，无收货地址 |
| S5 | 使用优惠券 | 基本流(1-5) + B4 + 基本流(6-8) | 有可用优惠券 |
| S6 | 支付失败重试 | 基本流(1-6) + B5 + 重新支付(7-8) | 使用余额不足的支付方式 |
| S7 | 订单超时 | 基本流(1-6) + B6 | 下单后30分钟不支付 |

## 五、判定表法

### 5.1 方法原理

判定表（Decision Table）适用于多个条件组合、多个动作的场景。通过穷举所有条件组合，设计测试用例。

### 5.2 实战案例

**案例：保险理赔判定**

```
条件：
C1: 事故类型 = 单方事故 / 双方事故
C2: 是否在保险期内 = 是 / 否
C3: 是否有有效驾驶证 = 是 / 否

动作：
A1: 全额理赔
A2: 部分理赔(70%)
A3: 拒赔
A4: 需要交警证明

判定表（条件组合规则矩阵）：
| 规则 | 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8 |
|------|---|---|---|---|---|---|---|---|
| C1单方/双方 | 单 | 单 | 单 | 单 | 双 | 双 | 双 | 双 |
| C2在保期内 | Y | Y | N | N | Y | Y | N | N |
| C3有效驾照 | Y | N | Y | N | Y | N | Y | N |
| A1全额理赔 | X |   |   |   |   |   |   |   |
| A2部分理赔 |   | X |   |   | X |   |   |   |
| A3拒赔     |   |   | X | X |   | X | X | X |
| A4交警证明 |   |   |   |   | X |   |   |   |

简化后：
- 规则1：单方事故+在保+有驾照 → 全额理赔
- 规则2：单方事故+在保+无驾照 → 部分理赔70%
- 规则3-4：不在保险期内 → 拒赔
- 规则5：双方事故+在保+有驾照 → 部分理赔+交警证明
- 规则6-8：双方事故+(不在保或无驾照) → 拒赔
```

## 六、正交试验法

当输入条件较多且每个条件有多种取值时，穷举所有组合不可行，使用正交表可以以最少的测试用例覆盖最多的条件组合。

```python
# 案例：Web页面兼容性测试
# 3个因素：操作系统(3种)、浏览器(3种)、屏幕分辨率(3种)
# 全量测试：3*3*3 = 27种组合
# 使用L9(3³)正交表：只需9种组合

# 因素水平表：
# A: 操作系统 = [Windows10, macOS, Ubuntu]
# B: 浏览器 = [Chrome, Firefox, Edge]
# C: 分辨率 = [1920x1080, 1366x768, 2560x1440]

# L9正交表（9个测试用例）：
test_combinations = [
    (Windows10, Chrome, 1920x1080),
    (Windows10, Firefox, 1366x768),
    (Windows10, Edge, 2560x1440),
    (macOS, Chrome, 1366x768),
    (macOS, Firefox, 2560x1440),
    (macOS, Edge, 1920x1080),
    (Ubuntu, Chrome, 2560x1440),
    (Ubuntu, Firefox, 1920x1080),
    (Ubuntu, Edge, 1366x768),
]
# 任何两个因素的每种水平组合都至少出现一次
```

## 七、错误推测法

错误推测法（Error Guessing）基于测试人员的经验和直觉，推测程序中可能存在的错误类型，有针对性地设计测试用例。这是对系统性设计方法的有力补充。

**常见易出错场景清单**：
1. **输入空值或null**：很多程序未正确处理null输入
2. **输入特殊字符**：单引号、双引号、`<script>`标签、SQL注入字符
3. **输入超长字符串**：可能导致缓冲区溢出
4. **并发操作**：同一用户多端操作、同一数据多人编辑
5. **边界时间**：月末、年末、闰年2月29日
6. **国际化场景**：多语言、多时区、特殊货币格式
7. **粘贴操作**：粘贴进来格式不一致的内容
8. **快速连续点击**：重复提交表单
9. **网络中断恢复**：操作中网络断开又恢复
10. **特殊数值**：0、负数、浮点精度问题(0.1+0.2 != 0.3)

## 八、测试用例编写规范

### 8.1 测试用例模板

```
用例编号: TC-[模块缩写]-[数字序号]  例: TC-ORDER-001
用例标题: [操作/场景] + [预期行为] 格式的肯定句
所属模块: 功能模块名称
优先级: P0(紧急)/P1(高)/P2(中)/P3(低)
测试类型: 功能/性能/安全/兼容性
前置条件: 执行用例前必须满足的条件

测试步骤:
  1. 从首页点击"我的订单"菜单
  2. 选择一条状态为"待支付"的订单
  3. 点击"立即支付"按钮
  4. 选择"微信支付"方式
  5. 点击"确认支付"

预期结果:
  1. 页面跳转到微信支付页面
  2. 显示订单金额和商品信息
  3. 扫码支付成功后，页面跳转到支付成功页
  4. 订单状态变更为"待发货"

测试数据: 订单号=20240115001, 金额=299.00
后置条件: 订单进入"待发货"状态
```

### 8.2 用例设计原则

1. **单一验证点**：一个用例尽量只验证一个功能点
2. **可重复执行**：避免依赖其他用例的状态
3. **清晰的预期结果**：用"是否"可以判断，不用"可能""大概"
4. **适度粒度**：不过于粗放（漏测多），也不过于细致（维护成本高）
5. **包含异常路径**：不要只设计Happy Path（正确路径）

## 九、总结

测试用例设计方法各有适用场景，实际工作中通常需要组合使用多种方法：
- **等价类划分法** + **边界值分析法**：适合有明确输入范围的字段验证
- **场景法**：适合业务流程复杂的端到端测试
- **判定表法**：适合多条件组合的业务规则
- **正交试验法**：适合多因素多水平的配置兼容性测试
- **错误推测法**：适合作为系统用例设计的补充

好的测试用例设计不仅要关注功能正确性，还要关注异常处理、边界条件和用户体验。测试用例是测试团队的宝贵资产，需要持续的维护和优化。
