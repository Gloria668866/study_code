# 数据仓库设计与建模

## 数据仓库概述

数据仓库（Data Warehouse）是一个面向主题的、集成的、相对稳定的、反映历史变化的数据集合，用于支持管理决策。由Bill Inmon于1991年提出这一概念。

### OLTP vs OLAP

| 特性 | OLTP | OLAP |
|------|------|------|
| 用途 | 日常事务处理 | 分析决策支持 |
| 数据量 | 小（GB级） | 大（TB-PB级） |
| 查询模式 | 简单CRUD | 复杂聚合查询 |
| 响应时间 | 毫秒级 | 秒到分钟级 |
| 数据时效 | 当前数据 | 历史数据 |
| 范式 | 3NF | 星型/雪花型 |

## 数据仓库分层

业界常见的数据仓库分层架构（以阿里巴巴分层模型为例）：

### ODS层（Operational Data Store）
原始数据层，保持源系统数据原貌，不做任何处理：
- 从业务数据库（MySQL/Oracle）同步数据
- 从日志系统采集行为数据
- 数据不做清洗转换，直接落地

### DWD层（Data Warehouse Detail）
明细数据层，对ODS数据进行清洗、标准化：
- 数据去重、空值处理、格式统一
- 维度退化和关联
- 保留最细粒度的事实数据

### DWS层（Data Warehouse Summary）
汇总数据层，按主题和维度进行轻度汇总：
- 按天/小时汇总交易金额、订单数
- 用户行为路径统计
- 构建宽表便于查询

### ADS层（Application Data Service）
应用数据层，面向具体业务场景：
- BI报表数据
- 推荐系统特征数据
- 风控模型输入数据

## 维度建模

### 星型模型
一个事实表连接多个维度表，维度表只有一层：
```
          ┌──────────┐
          │ 时间维度   │
          └─────┬────┘
                │
    ┌─────────┐ │ ┌──────────┐
    │ 产品维度 ├─┼─┤ 事实表    │
    └─────────┘ │ └─────┬────┘
                │       │
          ┌─────┴────┐  │  ┌──────────┐
          │ 地域维度   │  └──┤ 用户维度   │
          └──────────┘     └──────────┘
```

### 雪花模型
星型模型的扩展，维度表可进一步规范化，形成多级层次关系。查询时需要更多JOIN。

## 事实表与维度表

### 事实表
存储业务过程的度量值（通常是数值型）：
- **事务事实表**：每行对应一个业务事件（如一笔订单）
- **周期快照事实表**：按固定周期汇总（如每日余额）
- **累积快照事实表**：跟踪业务流程的多个里程碑

### 维度表
描述事实的属性信息：
- **缓慢变化维度（SCD）**：随时间变化的维度属性
  - SCD Type 1：直接覆盖（不保留历史）
  - SCD Type 2：新增行保留历史（最常用）
  - SCD Type 3：增加列保留上一次值

## 数据质量

### 质量维度
- **完整性**：数据记录是否完整
- **准确性**：数据值是否正确
- **一致性**：跨系统的数据是否一致
- **及时性**：数据是否按时更新
- **唯一性**：是否存在重复数据

### 质量监控
```sql
-- 空值检查
SELECT COUNT(*) FROM dwd_orders WHERE user_id IS NULL;

-- 重复检查
SELECT order_id, COUNT(*) FROM dwd_orders GROUP BY order_id HAVING COUNT(*) > 1;

-- 及时性检查
SELECT MAX(create_time) FROM dwd_orders;
```
