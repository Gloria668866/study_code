# 团队协作规范（必读）

> 5 人小团队、全栈制不分专职、按工单认领。本规范保证：**代码不乱、文档够细、记忆不丢、新人能接。**
> 配套：项目上下文看 [`AGENTS.md`](AGENTS.md) 和 [`PROJECT-MEMORY/`](PROJECT-MEMORY/README.md)；任务清单看 `docs/prd/开发任务拆解书.docx`。

---

## 1. 干完一件活，必须做的 4 步（DoD：完成的定义）

任何人做完一个工单 / 一块功能，**这 4 步缺一不可**，否则不算「干完」：

1. **更新仓库记忆** → 改 [`PROJECT-MEMORY/`](PROJECT-MEMORY/README.md) 里相关文件（决策、进度、坑），并更新 `04-交付物与进度.md`。
2. **写工作文档**（见第 4 节）→ 数据获取/处理的内容写进 **PRD-1**；其它写进 **自己模块的文件夹** `docs/工作记录/<模块>/`。**详细到刚学的人也能看懂**，必要时画图。
3. **规范提交代码**（见第 2 节）→ 有意义的 commit message。
4. **推到 GitLab** → push 到自己的分支，开 Merge Request（合并请求）等 review。

> 一句话：**改了东西 = 更新记忆 + 写文档 + 提交 + 推送**，四件套。

---

## 2. Git / GitLab 流程

### 分支策略（简单够用）
- `main`：只放可运行的稳定代码，**不许直接 push**。
- 干活开新分支：`feat/<工单号>-<简述>`，如 `feat/T7-text2sql`、`fix/crawl-retry`。
- 干完 push 分支 → 在 GitLab 开 **Merge Request** → 至少 1 人 review → 合并到 `main`。

### Commit message 规范
格式：`类型: 简述`（中文可以）。一次 commit 只做一件事。
```
feat: 实现 Text2SQL 自校验重试
fix: 修复增量采集当月数据未刷新
docs: 补充 PRD-1 清洗逻辑
chore: 加 .gitignore
```
类型：`feat`(新功能) `fix`(修bug) `docs`(文档) `refactor`(重构) `test`(测试) `chore`(杂项)。
> 提交结尾如需署名按团队约定；**禁止把 .env、密钥、大数据文件提交**（见第 5 节）。

### 常用命令
```bash
git checkout -b feat/T7-text2sql     # 开分支
git add -A && git commit -m "feat: ..."  # 提交
git push -u origin feat/T7-text2sql  # 推送
# 然后去 GitLab 网页开 Merge Request
```

## 3. 文档写在哪（重要规则）

| 你干的活属于 | 文档写到哪 |
|---|---|
| **数据获取 / 数据处理**（爬虫、清洗、入库、数据质量） | 写进 **PRD-1**（`docs/prd/`，找对应章节补，或让维护者补） |
| **其它任何模块**（Text2SQL、RAG、Agent、前端、接口、测试、部署…） | 写进 **自己模块文件夹** `docs/工作记录/<模块>/` |

`docs/工作记录/` 下已建好分模块文件夹（后端/前端/测试/运维），每个里有 `_模板.md`，复制改名即可。

## 4. 文档怎么写（详细到新人能懂）

**目标：一个刚学这行的人，照着你的文档能看懂你做了什么、为什么这么做、怎么跑起来。**

每篇工作文档至少包含：
1. **做了什么**（一句话概括 + 涉及的文件/函数）
2. **为什么这么做**（设计取舍，为什么不用别的方案）
3. **怎么运行 / 怎么验证**（命令、步骤、预期结果，能复制粘贴）
4. **输入 → 输出**（数据/参数进去，什么出来）
5. **踩过的坑**（报错和解决办法，省得别人重踩）
6. **必要时画图**：用 **Mermaid**（flowchart / sequence / ER），放代码块里：
   ````
   ```mermaid
   flowchart LR
     A[用户问题] --> B[Text2SQL] --> C[执行] --> D[图表]
   ```
   ````
   GitLab / VSCode / Typora 都能直接渲染 Mermaid。复杂图也可导出 PNG 放 `docs/工作记录/<模块>/img/`。

> 写文档别假设读者懂背景；专有名词第一次出现解释一句。代码贴关键片段并说明。

## 5. 环境与安全（底线）

- **密钥绝不入库**：LLM Key、DB 口令等只放 `.env`（已在 .gitignore）；仓库里只留 `.env.example`（占位）。
- **大数据/生成物不入库**：`data/raw/` 原始数据、`*.db`、`__pycache__/`、虚拟环境等已 gitignore。
- **Python 版本**：本机系统 Python 是 3.14 alpha，**很多包装不上**。开发统一用 **Python 3.12**；爬虫用 scrapling 专用 venv（见 `PROJECT-MEMORY/03`）。
- 改了依赖 → 更新 `requirements.txt`。

---

## 6. Code Review 要点（合并前看这些）
- 能跑通、有对应文档、记忆已更新。
- 没有硬编码密钥、没有提交大文件。
- 命名清晰、关键逻辑有注释。
- 改了行为 → 有对应测试或验证说明（见测试工单 T13）。

---

## 7. 新人上手三步
1. 读 `AGENTS.md` + `PROJECT-MEMORY/` 全部 → 懂项目。
2. 读 `docs/prd/` 三份文档 → 懂设计；读 `docs/工作记录/<你的模块>/` → 懂现状。
3. 从 `开发任务拆解书.docx` 认一个工单，按本规范开干。
