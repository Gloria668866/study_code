# 车市镜 · 吃透项目的学习路线（源码 + 流程图）

> **怎么用这份**：从上往下走，**别按文件名字母序读、要按「数据怎么流」读**。每一步给了①读哪些文件 ②配哪份文档/图 ③读完要能回答什么（自检）。
> **一句话心法**：这个项目的灵魂是 `app/graph.py`——它是一张地图，把两个脑（Text2SQL 数据脑 + RAG 知识脑）编排起来。**先看懂这张图，再下钻每个节点**，比从任何别的地方切入都快。

---

## 总览：这是个什么项目（30 秒）

车市镜 = 新能源汽车市场情报**对话式 Agent**。用户问一句话，系统判断意图后走两个脑之一或并行：
- **数据脑（Text2SQL）**：自然语言 → SQL → 查销量库 → 出图表 + 洞察。
- **知识脑（RAG）**：自然语言 → 检索知识库（研报/政策/口碑）→ 带引用作答、防幻觉。
- **编排（LangGraph）**：状态图把两个脑、重试环、并行合并、澄清/闲聊兜底串起来。
- **外壳**：FastAPI（SSE 流式）+ Vue3 前端 + PostgreSQL/pgvector + Celery + docker-compose 部署。

## 第 1 步 · 跟着「一个请求」走完整条链路 ★最重要（1 天）

**这是整个学习路线的核心**。不要孤立看文件，要跟着一个问题（比如「2025年纯电销量Top10」）从前端一路走到出图。

**正向链路（请求怎么进来、答案怎么出去）**：
1. 前端发问：`frontend/src/components/Composer.vue` → `composables/useChat.js` → `api/events.js`（**原生 fetch+ReadableStream 解析 SSE**，不是 EventSource——见复盘 B5）
2. 后端入口：`app/main.py` 的 `POST /api/ask` → 调 `stream_agent()`（SSE 渐进推送，用工作线程跑图 + 线程安全队列桥接）
3. 编排核心：`app/graph.py` 的 `run_agent / stream_agent` → `build_graph()` 编译的状态图
4. 沿状态图走（**先只看 sql 这条路**）：`intent_router → schema_link → gen_sql → exec_sql → verify_sql → chart → insight → compose → END`
5. 每个节点把决策写进 `State.trace` → `main.py` 把字段变成 SSE 事件（intent/sql/rows/chart/insight/citation/done）→ 前端双脑分开渲染

**先读这一份**：`app/graph.py` **文件顶部的 ASCII 状态流转图 + docstring**——它就是这张地图本身，5 分钟看懂整体骨架，再去看每个节点函数。

**读完要能回答**：
- 一个问题进来，依次经过哪些节点？trace 里能看到什么？
- SSE 为什么用「节点级快照」流式推、`emitted` 集合是干嘛的（每种事件只推一次）？
- 为什么澄清/闲聊是 `→END`，而 sql/rag/hybrid 最后汇到 `compose`？

## 第 2 步 · 吃透数据脑 Text2SQL（1~2 天）

**按数据流读**（prompt 组装 → 生成 → 安全 → 执行 → 选图）：
| 文件 | 看什么 |
|---|---|
| `app/text2sql.py` | **Prompt 工程**：`DOMAIN`（把星型模型语义/枚举/口径喂给 LLM）+ `FEWSHOT`（示例）+ `SYS`（规则）；`_extract_sql`（从 LLM 输出抠 SQL） |
| `app/schema_linking.py` | 表多时先挑相关表只喂相关 schema（现在是关键词版，可升级向量版） |
| `app/sql_guard.py` | **安全护栏**：sqlglot 解析 AST、只放行单条 SELECT、`find_all` 揪出子查询里藏的写操作（**面试加分，必懂**） |
| `app/db.py` | 只读执行 + schema introspection（生产用只读账号 = 第二道防线） |
| `app/charts.py` | **规则引擎选图**（不用 LLM）：看结果集结构判 bar/hbar/line/pie/table |
| `app/graph.py` 的 `gen_sql/exec_sql/fix_sql/verify_sql` 节点 | 把上面这些串成「生成→执行→报错回喂修正→语义自校验」的闭环 |

**配套文档**：`docs/面试深度理解手册.md` 的 Text2SQL 模块 + PRD-2 的 Text2SQL 章。

**读完要能回答**：
- DOMAIN 为什么要手写（schema introspection 只给类型、给不了业务含义）？
- 护栏为什么用 AST 而不是正则/关键字黑名单？怎么防 `SELECT 1; DROP TABLE`？
- 自校验重试环治什么、治不了什么？语义自校验（`verify_sql`，新增）补的是哪一块？（见复盘 A3）

## 第 3 步 · 吃透知识脑 RAG（2 天）★面试最重，深挖最多

**分两条链读**：

**A. 离线入库管线**（文档怎么变成可检索的向量）：
`app/rag/ingest.py`（编排）→ `store.py`（MinIO 存原件）→ `parse.py`（PDF/HTML 解析）→ `chunk.py`（**结构感知父子分块**，small-to-big）→ `embed.py`（BGE-large-zh 算 1024 维向量）→ `pg.py`（写 PostgreSQL+pgvector，HNSW 索引）。异步：`app/rag/tasks.py` + `app/celery_app.py`。

**B. 在线检索-生成**（问题怎么变成带引用的答案）：
`app/rag/retrieve.py` 一个文件讲完全链路：
`hybrid_recall`（向量 + 全文 **RRF 融合**）→ `rerank`（bge-reranker 精排）→ `merge_parents`（**父块归并四情形 A/B/C/D**）→ `generate`（拼来源头 + 强约束 Prompt → 带引用 JSON + 防幻觉 + 防注入）。
辅助：`app/rag/text.py`（**jieba 中文分词**——修全文检索召回 0 的那个，见复盘 B1）。

**配套文档**：`docs/面试深度理解手册.md` 的 RAG 模块 + `PROJECT-MEMORY` 里「RAG 切块与检索」设计决策 + PRD-2 §5。

**读完要能回答**：
- 为什么「检索子块、生成换父块」（small-to-big）？子块多大、为什么是 512 token 这个硬约束？
- 混合召回为什么要 RRF？reranker（cross-encoder）和向量召回（双塔）的区别？
- 子块命中多个父块的四情形（同父/相邻/互补/冲突）各怎么处理？为什么冲突不取平均？
- 防幻觉两道闸是什么（召回空 / 重排分低于阈值）？多租户为什么要「预过滤」？（见复盘 B6）
- **B1 那个中文分词闭环故事**（recall 0.46→0.83）——这是你 RAG 最硬的故事，必须能完整讲。

## 第 4 步 · 回到编排层 LangGraph 通读全图（1 天）

第 1 步只看了 sql 一条路，现在把 `app/graph.py` **整张图**吃透：
- `AgentState`（TypedDict）字段设计；`trace` 为什么用 `Annotated[list, operator.add]`（并行分支合并）
- 五类意图路由：sql / rag / hybrid / clarify / **chat（新增闲聊兜底）**
- **hybrid 并行**：条件边返回列表 fan-out 两条脑链；**`compose` 用 `defer=True` 做 join**（不然两链长度不一 compose 跑两次，见复盘 B2）
- 两个环：执行报错重试环（`exec_sql ⇄ fix_sql`）+ 语义自校验环（`verify_sql → fix_sql`）
- 为什么用状态图而非 if-else（顶部 docstring 已写）；关于 checkpointer/interrupt 的取舍（见复盘 A2）

**配套**：`docs/工作记录/后端/2026-05-25-LangGraph双脑编排.md`（含状态图）。

**读完要能回答**：画出整张状态图、说清每条边的触发条件、并行合并和重试环怎么终止。
