### A1.【活 Bug】Text2SQL 的 `score` 口径过时——prompt 把模型骗瘸了

- **症状**：用户问「口碑评分最高的 10 个车系」，模型永远查不出来 / 不敢用 `score` 排序。
- **根因**：`app/text2sql.py` 的领域提示（DOMAIN）写死了 `score 评分[恒为NULL]`、`score 恒为 NULL，不要用它排序`。但后来补采口碑详情回填后，`fact_review.score` 已有 **5858 条真实评分 [0,5]**。**prompt 没跟着数据更新，等于用一句过时的话把模型的能力锁死了**。是数据质量评测（18 项 expectation）主动发现的。
- **修复**：把口径改成 `score 口碑评分[0-5分，部分车系为NULL]`，并加一条规则——按口碑排序/筛选前必须 `WHERE score IS NOT NULL`（防 NULL 混入），再补一条口碑 few-shot 示例固化这个写法。
- **面试怎么讲**：「我有一套数据质量评测（行数/非空/值域/唯一/外键/口径 18 项），它发现我 prompt 里的 `score 恒为 NULL` 和数据库实际不一致——这是**LLM 应用特有的坑：prompt 是另一份'隐式 schema'，数据变了 prompt 不变就会产生静默错误**。我靠评测把它抓出来并修了。」
- **会被追问**：
  - Q：你怎么保证 prompt 里的口径和库一直对齐？A：把口径检查纳入数据质量评测红线（CI 里跑），数据回填/变更触发复检；更进一步可以从 schema 注释自动生成部分 DOMAIN。
  - Q：score 有 NULL，AVG 会怎样？A：SQL 的 AVG 自动跳过 NULL，但如果不加 `WHERE score IS NOT NULL`，ORDER BY 时 NULL 的排序行为依方言而定（SQLite 里 NULL 最小），会污染 Top 榜，所以显式过滤。

### A2.【信任漏洞 ★最重要】docstring 宣称「挂起等人重入」，但代码里根本没有 checkpointer

- **症状**：`app/graph.py` 开头写「支持挂起等人——clarify 节点反问后 END，用户回答再重入」。但 `run_agent()` 每次都是 `_GRAPH.invoke(全新 init)`，`build_graph()` 编译时**没有 checkpointer、没有 thread_id**。
- **为什么是致命伤**：这类「叙事 > 实现」的缝，比技术深度更要命——面试官一句「你这个挂起重入，LangGraph 的 checkpointer 用哪个？`MemorySaver` 还是 `PostgresSaver`？thread_id 怎么传？」就能让你当场答不上来，**进而怀疑你整份简历的真实性**。
- **修复**：没有去硬塞一个 checkpointer（那样反而会引入新 bug，见下），而是**把叙事改成和实现一致**，并主动写清取舍：
  - 准确描述：clarify 是**对话级澄清**——反问后结束本轮，用户的补充作为带 `history` 的**新一轮请求**进来，`intent_router` 借 history 理解指代后重新路由；状态持久化在 `message` 表 + 前端会话，而非 LangGraph 检查点。
  - 主动给出「为什么不用 checkpointer」：再叠一层 LangGraph 检查点是重复持久化，且 SSE 流式下管理恢复点复杂度高、收益低。**何时该上**：要做图内多轮挂起（分步收集多个查询槽位、Human-in-the-loop 审批）时，才引入 `checkpointer + thread_id=conversation_id`。
- **为什么不硬加 checkpointer**（这本身是个坑）：现在 `clarify → END`。若天真地 `compile(checkpointer=MemorySaver())` 并用 `conversation_id` 当 thread_id，**正常的追问会去「恢复」一个已经跑到 END 的图**，把后续问答弄坏。真要做图级挂起，正确姿势是 `interrupt()`（动态中断）+ 用 `Command(resume=...)` 恢复，那是另一套流程，不是加一行参数。
- **面试怎么讲**：把它讲成**主动的架构取舍**——「LangGraph 提供 checkpointer + interrupt() 做图内断点恢复，我刻意没用，因为我的澄清状态本来就落在 message 表和前端会话里，再叠一层是重复造轮子。如果将来要做 Human-in-the-loop 审批或多槽位收集，我会用 PostgresSaver + thread_id=conversation_id。」——**会用 ≠ 滥用，知道什么时候不用才是工程成熟度。**
- **会被追问**：
  - Q：interrupt() 和 interrupt_before 区别？A：`interrupt_before/after` 是在编译期指定某节点前/后静态中断；`interrupt()`（新版）是节点内动态抛出中断、把控制权交回调用方，恢复时从断点继续，更灵活。
  - Q：那你多轮上下文怎么传的？A：`_load_history` 取最近 6 条消息 → `_history_block` 压成一小段塞进 intent_router 和 gen_sql 的 prompt，让模型理解「那丰田呢」这种指代。

### A3.【技术加分 ★头牌】Text2SQL 语义自校验——治「能跑但答非所问」的那 22%

- **背景痛点**：实测 Text2SQL 执行准确率 **78.3%**（60 题，按执行结果集等价比对）。原来的自校验重试环（`exec_sql ⇄ fix_sql`）**只能治「执行报错」**——把 SQL 错误回喂模型修语法。但**「能跑通但语义错」（口径选错、过滤漏了、聚合维度不对）重试环救不了**，因为它根本不报错。这是那 22% 的主要构成。
- **修复**：新增 `verify_sql` 节点，把语义错也纳入闭环：
  - 流转：`exec_sql 成功 → verify_sql（让 LLM 核对「结果是否真的回答了问题」）→ 匹配则 chart，不匹配则把原因回喂 fix_sql 重生成`。
  - **三个工程化考量（讲出来很值钱）**：
    1. **FAIL-OPEN**：开关关闭 / 空结果 / 重试预算耗尽 / 校验自身抛异常 —— 一律放行。**新增的校验绝不能让系统比不校验更差**。
    2. **开关可关**（`SEMANTIC_CHECK=on/off`）：它每条成功查询多花 1 次 LLM 调用（+延迟+成本），demo 想要快可一键关。**这就是「准确率 vs 延迟/成本」的取舍，要会主动说。**
    3. **复用现成重试环**：不匹配时把「为什么不匹配」当 error 喂给 `fix_sql`，共用 `MAX_SQL_RETRY` 预算上限，**保证一定收敛**（verify 在预算耗尽时直接放行，不会死循环）。
- **面试怎么讲**：「我诚实承认我的重试只治执行报错、治不了语义错——这是 22% 错误的大头。所以我加了一道语义自校验：执行成功后让 LLM 看着真实结果回答'这个结果对得上问题吗'，不对就带着原因重生成。它是 fail-open + 可开关的，因为我清楚它拿延迟和成本换准确率。」
- **会被追问**：
  - Q：用 LLM 校验 LLM，它自己也会错啊？A：对，所以它只做**保守拦截**（明确判 mismatch 才回退），且 fail-open；它不是真值，是「再看一眼」的廉价二次确认。更稳的做法是加**确定性校验**——比如检查用户问题里的品牌/车系实体是否都出现在 SQL 的 WHERE 里（我 prompt 里已有这条约束，可以提到节点里做硬校验）。
  - Q：为什么不用 self-consistency（多次采样投票）？A：那个成本更高（N 倍调用），对延迟敏感的对话场景不划算；我选了单次保守校验 + 可关开关。
  - Q：怎么证明它真有用？A：在评测集上做 A/B（开/关各跑一遍 60 题 EX），对比准确率增量 vs 延迟增量——**这正是下一步要补的数字**（诚实边界，见文末）。

### A4.【工程意识 ★稀缺】LLM 调用埋点——「一次查询多少钱、P95 多慢」90% 应届生答不出

- **症状**：整个项目原本**没有任何 token 成本 / 延迟可观测**，被问「一次查询多少钱、P95 延迟多少」只能现编。
- **修复**：在 `app/llm.py` 的 `chat()` 里加埋点——每次调用记录耗时 + token 用量（读 OpenAI 响应的 `usage`），`get_metrics()` 输出 **P50/P95/P99 延迟、平均 token、估算成本**（单价按 `.env` 配）。
  - **线程安全（这个细节体现你真懂）**：图是丢到**工作线程**里跑的（见 `main.py` 的 `worker()`），多线程并发累加指标，所以用 `threading.Lock` 保护，延迟用 `deque(maxlen=1000)` 存最近 N 次算分位。
  - 分位数用线性插值，离线单测验证过（不联网）。
- **面试怎么讲**：「我给 LLM 调用加了埋点，能直接给出 P50/P95/P99 延迟、每问平均 token 和估算成本。这是上线降本的依据——比如我能看到哪类问题最贵，再决定要不要上**语义缓存**（相似问题命中缓存不打 LLM）。埋点本身要注意线程安全，因为我的图在工作线程跑。」
- **会被追问**：
  - Q：流式（chat_stream）的 token 怎么统计？A：流式默认最后一个 chunk 才带 usage（需开 `stream_options={"include_usage":True}`），目前只埋了同步 `chat()`（意图/SQL/修正/洞察/RAG 生成都走它，是成本主路径），流式埋点是待补项。
  - Q：P95 用 deque(1000) 够吗？A：滑动窗口近似，够看趋势；要精确历史分布就上时序库（Prometheus histogram / Langfuse）。
  - Q：怎么接到线上？A：挂一个 `/api/metrics` 端点，或每个请求结束 log 一行，或直接接 Langfuse 把 trace（我的 `AgentState.trace` 已经埋好全量数据）打出去。

### A5.【测试纪律】改了图拓扑，测试必须同步改 + 补新路径覆盖

- **症状**：把 `route_exec` 成功分支从 `chart` 改成 `verify_sql` 后，老测试 `test_route_exec_success_to_chart`（断言返回 `"chart"`）失效。
- **根因**：测试是**契约的固化**。我故意改了契约（加了一道校验），测试就该跟着改——这不是「改测试迁就代码」，而是「契约变更要在测试里显式登记」。
- **修复**：① 把该用例改名 `test_route_exec_success_to_verify` 并更新断言；② **补齐新路径覆盖**——`route_verify` 的 ok→chart / mismatch→fix_sql / 缺字段 fail-open，以及 `verify_sql` 在「空结果」「预算耗尽」时不调 LLM 直接放行。15/15 全过。
- **面试怎么讲**：「我改图拓扑时连带更新了路由单测，并给新增的校验节点补了 fail-open 的确定性测试——这些测试不调 LLM、纯函数、CI 里能稳定跑。」
- **会被追问**：Q：你怎么测一个会调 LLM 的节点？A：把**纯路由逻辑（条件函数）**和**带 LLM 的节点**分开测——路由函数纯函数化、给定 state 断言走向；LLM 节点只测它的 fail-open 分支（开关关 / 无数据 / 异常）这些不触发网络的路径，LLM 真实效果交给评测台（60 题 EX）兜。

### A6.【真实 Bug ★用户实测撞到】问"你吃饭了吗"，它直接给我画了张分析图

- **症状**：对话框里问一句和车毫无关系的"你吃饭了吗"，系统**当成数据查询，跑了 Text2SQL 还画了图**。
- **根因（一个 bug，三层原因）**：
  1. **没有"闲聊/超纲"意图**：`intent_router` 只认 sql/rag/hybrid/clarify 四类，问候/闲聊/与车无关的问题**无处可去**。
  2. **LLM 兜底默认瞎猜 sql**：分类兜底是 `next((...), "sql")`——LLM 没吐出明确意图词时**默认 sql**，于是闲聊被拖去 schema_link→gen_sql→exec→chart。
  3. **关键词规则子串误命中**：朴素 `any(k in q)` 太泛——`"多少"`命中"你多少岁/这个多少钱"、`"对比"`命中"对比心情"、`"怎么样"`命中"今天天气怎么样"，**这些根本到不了 LLM 就被规则直接判错**。
- **怎么找到的**：写了个**离线测试炮台**，拿 8 句闲聊 + 攻击 SQL + 边界数据集中打意图规则/SQL 护栏/图表/抽取四个模块。意外收获：**SQL 护栏满分**（strftime/`||`/CTE/OFFSET 合法查询全过，多语句/DROP/ATTACH/PRAGMA/CTE 藏 DELETE 全拦），bug 全集中在意图路由。
- **修复（系统性，不是补丁）**：
  1. **新增 `chat` 意图 + `chitchat` 节点**：友好说明能力边界、引导回车市话题，`→END`，**不查数据不出图**。
  2. **关键词瘦身**：把会在闲聊里出现的泛词（多少/对比/怎么样/为什么/原因）移出快速通道，只留强领域词（销量/排名/环比/Top…），泛词交给 LLM 判。
  3. **闲聊短路**：明显问候词（你好/吃饭/笑话/你是谁…）且无领域信号 → 直接 chat，连 LLM 都不调（省成本）。
  4. **安全默认**：LLM 兜底默认从 `sql` 改成 **`clarify`**——真分不出来时**反问，而不是瞎猜去查数据**。
- **面试怎么讲**：「我做意图识别是'规则前置省成本 + LLM 兜底保准确'。但踩过一个坑：规则用子串匹配太贪婪、又缺一个'闲聊/超纲'类别、兜底还默认猜 sql，导致'你吃饭了吗'被画成图。我的修法是**系统性的**——补 chat 意图 + 规则只留强领域词 + 兜底默认改 clarify（分不出就反问）。**一个对外产品必须优雅处理'超出能力范围'的输入，这是鲁棒性的基本功。**」
- **会被追问**：
  - Q：为什么不干脆全用 LLM 分类？A：规则前置省成本省延迟（销量/排名这类一眼是 sql 的没必要调模型），但规则只能放**无歧义的强信号**，模糊的让 LLM 兜——这是准确率与成本的平衡。
  - Q：怎么知道一句话是"超纲"而不是"信息不足"？A：chat=与车市无关（你好/讲笑话）；clarify=与车市相关但口径不清（"哪个车好"——好指销量还是口碑？）。我在 prompt 里把这俩的区别写清了。
  - Q：还有什么没收尾的？A：前端意图徽章还没有"闲聊"这个 label（新 intent 值），会显示原始值——是个小的前端跟进项（不影响答案正文渲染）。

### B1.【★必背·完整闭环】RAG 全文检索对中文不分词 → 中文 query 召回 0

- **症状**：RAG 评测基线 recall 只有 **0.455**，乘联会(0.28)/研报(0.30)这类「政策/数字」文档召回垮，而口碑(0.75)还行。
- **根因**：混合召回里的全文检索路用 PostgreSQL `to_tsvector('simple', ...)`，**`simple` 分词器对中文不切词**——把整句中文当一个 token，中文 query 几乎匹配不上 → 全文路召回≈0 → **混合召回退化成纯向量**（评测消融里 `vec==hybrid` 0.75 就是铁证）。只有 '800V' 这种纯英数才有效。
- **修复**：应用层 jieba 分词——新增 `content_tokens` 列存切好的词，查询时也对 query 切词拼成 OR 的 tsquery（`app/rag/text.py` + `pg.py`）；存量 28629 chunk 用回填脚本处理，**只补分词列、不重算向量**（省算力）。
- **数字（背下来）**：recall **0.455→0.830**、乘联会 0.28→0.89、研报 0.30→0.85、跨段 0.17→0.92、correct 0.42→0.80、faith 0.34→0.71，abstain（拒答）保持 1.0。
- **面试怎么讲**：「评测台先发现指标异常（recall 低 + 消融里混合召回竟然等于纯向量）→ 我顺着'为什么混合召回没增益'查到中文分词 → jieba 修复 + 存量回填 → 同一评测集回归，recall 0.46→0.83。**这是一条评测驱动的完整闭环**，也是我做 RAG 不是调包、而是有度量有归因的证据。」
- **会被追问**：Q：为什么不用 pg_jieba 扩展？A：应用层分词不依赖 DB 扩展，本地 SQLite / 换库都能跑，部署简单；pg_jieba 是生产可选的性能优化。Q：为什么消融能帮你定位？A：消融把「纯向量 / 混合 / +rerank」拆开比，发现混合==纯向量就说明全文路没贡献，直接指向召回侧。

### B2.【并行编排经典坑】hybrid 模式 compose 被触发两次 → `defer=True`

- **症状**：hybrid（数据脑 + 知识脑并行）下，汇总节点 `compose` 跑了两次，答案拼重复。
- **根因**：两条脑链长度不一，先到的分支就触发了 compose，后到的又触发一次（扇入节点没等齐所有在途分支）。
- **修复**：`g.add_node("compose", compose, defer=True)`——`defer=True` 让它延到所有在途分支都完成才跑一次（join 语义）。
- **面试怎么讲**：「这是并行图'扇出-扇入'的典型坑——fan-in 节点要 defer 到所有上游完成才执行，否则会被每条先到的边各触发一次。体现我是真用状态图做并行，不是摆样子。」

### B3.【依赖地狱】passlib 1.7.4 与 bcrypt ≥4.1 不兼容 → 钉 `bcrypt==4.0.1`

- **症状**：登录注册时 bcrypt 报错（自检 >72 字节抛异常）。
- **根因**：passlib 1.7.4 调用 bcrypt 的方式与 bcrypt 4.1+/5.x 的 API 不兼容；且 bcrypt 本身有 **72 字节密码上限**。
- **修复**：`requirements.txt` 钉死 `bcrypt==4.0.1`。
- **面试怎么讲**：「鉴权用 bcrypt（加盐慢哈希）+ JWT。踩过 passlib 和新版 bcrypt 不兼容的坑，钉版本解决；也知道 bcrypt 有 72 字节上限，超长密码要先预哈希。」

### B4.【前后端契约 Bug】SSE 的 rows 发「字典行」，前端按「列位」取值 → X 轴全 undefined

- **症状**：live 模式下图表 X 轴全是 `undefined`、没有柱子。
- **根因**：后端 `rows` 推的是「字典行」（`{col: val}`），前端按数组下标 `r[i]` 取值需要「数组行」——**前后端对 rows 形态的约定不一致**。
- **修复**：后端 rows 事件对齐 PRD §9.1，统一发 `{columns, rows(二维数组)}`，前端零改动即正确。
- **面试怎么讲**：「这是典型的前后端契约不一致——同一个 rows，一边以为是字典一边按数组取。教训是**SSE/接口契约要写死并双方对齐**（我后来专门做了一版 §9.1 协议对齐，前端撤掉了临时的归一化兼容层）。」

### B5.【原生 API 限制】`EventSource` 只支持 GET，而 `/api/ask` 是 POST

- **症状**：想用浏览器原生 `EventSource` 收 SSE，但提问接口是 POST（要带 body + Bearer token）。
- **根因**：原生 `EventSource` 只能 GET、不能自定义 header。
- **修复**：前端用 `fetch` + `ReadableStream` 自己解析 SSE 流，从而能 POST、能带 `Authorization`。
- **面试怎么讲**：「流式我没用现成 EventSource，因为它只支持 GET 不能带 token；我用 fetch 读 ReadableStream 手写 SSE 解析，支持 POST + 鉴权头。」

### B6.【RAG 细节坑】CrossEncoder reranker 别二次套 sigmoid + 多租户要预过滤

- **症状①**：reranker 分数不对劲。**根因**：`bge-reranker` 的 CrossEncoder 已自带 sigmoid，外面又套了一层 → 分数被压。**修复**：去掉二次 sigmoid。
- **症状②**：多租户下偶发召回不足 / 担心串户。**根因**：HNSW 是近似检索，若「先检索再按 user_id 过滤」（后过滤）会导致 Top-K 被别人的数据占掉、自己的召回不足。**修复**：把 `WHERE user_id AND deleted_at IS NULL` **预过滤**进 SQL，召回阶段就只在本租户范围里找。
- **面试怎么讲**：「两个 RAG 工程细节：reranker 自带 sigmoid 不能二次套；多租户必须**预过滤**而不是后过滤，否则近似检索会让召回不足还可能串户。」

### B7.【异步坑】Celery `.delay()` 默认等 result backend 阻塞 → `task_ignore_result=True`

- **症状**：触发异步解析任务时调用方被阻塞。
- **根因**：Celery `.delay()` 默认要把结果写 result backend，配置不当会阻塞。
- **修复**：`task_ignore_result=True` 做 fire-and-forget（解析进度靠 `kb_document.status` 轮询，不靠 Celery 结果）。
- **面试怎么讲**：「文档解析（解析→分块→向量化）耗时，丢 Celery 异步；用 fire-and-forget + DB 状态轮询（parsing→ready/failed），不依赖 Celery result backend，避免阻塞。」

### B8.【ORM 坑】`create_all` 不会改已存在的表 → 加列要手动 ALTER

- **症状**：给 `Message` 加 `result_meta` 列后，PG 里查不到这列、报错。
- **根因**：SQLAlchemy `create_all` 只建不存在的表，**不会给已存在的表加列**。
- **修复**：手动 `ALTER TABLE` 补列（生产应上 Alembic 迁移）。
- **面试怎么讲**：「`create_all` 只适合冷启动建表，schema 演进要用迁移工具（Alembic）。我踩过加列没生效的坑，手动 ALTER 了，正解是引入迁移。」
