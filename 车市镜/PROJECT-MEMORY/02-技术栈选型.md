# 02 · 技术栈选型（含「为什么」和被否方案）

> 全部为 2026-05-22 与用户敲定的锁定项。

| 层 | 选型 | 为什么 | 被否决 |
|---|---|---|---|
| 爬虫 | **Scrapling** 0.4.8 | 一个库集成抓取+反爬绕过(StealthyFetcher)+自适应解析，替代三件套 | Scrapy/Playwright/httpx 三件套（太重） |
| 文档解析 | **MinerU（CPU 模式）** | PDF→Markdown，复杂版面/表格抽取强；CPU 够用不买 GPU | pdfplumber/unstructured（弱） |
| 对话大模型 | **DeepSeek-V4** | OpenAI 兼容、中文强、性价比高；config.py 可切换 | 通义 qwen（备选保留） |
| Embedding | **BGE-large-zh（本地，1024维）** | 中文检索好、零成本、可离线 | 用对话模型做向量化（错误做法，禁止） |
| 原始数据存储 | **MinIO 对象存储** | GB 级数据，自建 S3 兼容、轻量、可重放 | ❌ **Hadoop**（TB~PB 级才用，5人团队过度设计） |
| 结构化库 | **PostgreSQL**（本地 demo 用 SQLite） | SQL 强、运维近零；装 pgvector 同库存向量 | 大数据组件（用不上） |
| 向量库 | **pgvector（装在同一个 PG 里）** | 少一个组件，与业务库同栈 | Chroma 仅本地 demo 起步、Milvus 规模大才用 |
| 队列/缓存 | **Redis** | 爬虫调度、任务队列、结果缓存 | — |
| 编排 | **LangGraph + LangChain** | 支持环(重试)/挂起(澄清)/并行(双脑)/可观测，胜过 if-else | 裸 if-else |
| SQL 护栏 | **sqlglot** 解析 AST | AST 级判断语句类型，比正则可靠 | 正则 |
| Rerank | **bge-reranker** | 提升 RAG Top-K 命中精度 | — |
| 后端 | **FastAPI**（SSE 流式） | 异步、SSE 友好、自动文档 | — |
| 前端 | Vue 3 + Vite + ECharts + EventSource | **产品门面，后续要上线，按真实产品标准做**：体现 BI Agent 产品逻辑（双脑分渲、Agent 思考过程、情报感、示例引导）；组件化+API base 走配置 | — |
| 测试/评估 | **pytest + RAGAS + DeepEval(CI) + Great Expectations** | RAGAS 专测 RAG；DeepEval 接 CI 回归；Text2SQL 另做执行结果比对评测集 | — |
| 部署 | **单台云 VPS 自托管全栈** · Docker + docker-compose + Caddy(HTTPS) | 初期最省钱、运维可控 | 云托管(贵)、物理机(重) |

## 重要原则
- **Embedding 模型 ≠ 对话模型**：向量化用 BGE，绝不用 DeepSeek。
- **坚决不上 Hadoop**：真到 TB 级再考虑 ClickHouse/数据湖。
- **引擎 schema-agnostic**：换行业只换「领域包」目录（种子数据 + 口径词典 + 同义词 + few-shot + 初始语料）。
